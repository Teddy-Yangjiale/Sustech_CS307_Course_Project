## AviationDataLoader.java

- 导入前确保：性别有三类，海拔可以为空

  

- | **级别**   | **导入策略**                 | **核心机制**                                                 | **优势**                                                     | **劣势**                                                     | **10万条数据耗时预估** |
  | ---------- | ---------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ | ---------------------- |
  | **级别 1** | 逐条插入 + 自动提交          | 每读一行执行一次 `executeUpdate`，数据库自动开启并提交事务。 | 代码实现最简单。                                             | 网络通信极度频繁；数据库每行都要进行物理磁盘日志写入（I/O灾难）。 | 30 - 60 分钟           |
  | **级别 2** | 逐条插入 + 手动事务          | 关闭自动提交。在内存中完成数据处理，最后统一 `commit`。      | 解决了磁盘频繁写入的瓶颈。                                   | 依然需要逐行发送网络请求，网络 I/O 瓶颈未解决。              | 1 - 3 分钟             |
  | **级别 3** | **JDBC 批处理** (本项目方案) | 内存打包 `addBatch`，定量定频发送 `executeBatch`，最后统一提交。 | 解决网络与磁盘双重瓶颈；保留了代码层面对异常数据（如空值、格式转换）的控制力。 | 需要在代码中维护分批发送逻辑，防止内存溢出。                 | 3 - 5 秒               |
  | **级别 4** | 数据库底层 `COPY`            | 绕过 SQL 解析引擎，直接将格式化文本以二进制流形态灌入底层存储。 | 达到数据库写入的物理极限速度。                               | 极度脆弱，遇到任何格式、约束错误或空值会直接崩溃回滚；丧失代码层面的业务处理能力。 | < 0.5 秒               |

## Client.java

目前已经支持project中所有基本功能：

| **功能模块**         | **核心操作的表**                | **SQL 动作属性**                      | **系统角色**           |
| -------------------- | ------------------------------- | ------------------------------------- | ---------------------- |
| **Generate Tickets** | `flight`, `flight_cabin`        | 全局只读聚合 (`SELECT SUM/COUNT`)     | 报表系统 (BI)          |
| **Search Flights**   | `airport`, `route`, `flight`... | 高频复杂联查 (`SELECT JOIN`)          | 检索引擎 (Search)      |
| **Manage Orders**    | `ticket_order`, `flight_cabin`  | 高危写操作 (`INSERT/UPDATE` + 事务锁) | 交易核心 (Transaction) |

### 全局架构：JDBC 与 CLI 交互层

**1. 逻辑实现：**

- 使用 `Scanner` 配合 `while(true)` 构建了一个经典的 REPL（读取-求值-输出-循环）命令行菜单。
- 全篇使用 `PreparedStatement` 替代了普通的 `Statement`。

**2. 为什么这么实现？**

- **防 SQL 注入**：`PreparedStatement` 通过 `?` 占位符预编译 SQL 语句，严格区分了“代码”和“数据”，从根本上杜绝了类似于 `1=1 OR drop table` 这样的 SQL 注入攻击。
- **复用性能**：预编译的 SQL 会被数据库引擎缓存执行计划，多次执行同样的查询时速度更快。

**3. 替代方案（工业级进阶）：**

- **连接池 (Connection Pool)**：目前的做法是全局持有一个 `Connection` 实例。如果网络波动断开，程序就会崩溃；如果在多线程下，这个单例会成为瓶颈。企业级开发通常会引入 **HikariCP** 或 **Druid** 连接池，系统启动时初始化 10 个连接放着备用，随取随还。
- **ORM 框架**：在实际的 Java 后端开发中，我们很少手写这么长的 SQL 字符串，而是使用 **MyBatis** 或 **Hibernate / Spring Data JPA**。它们可以自动把数据库的行映射成 Java 对象（OOP 思想），消除大量模板代码。

### 模块一：`generateTickets` (聚合统计)

**1. 逻辑实现：**

利用 `SUM(CASE WHEN fc.cabin_class = '...' THEN fc.remain_count ELSE 0 END)` 语法，在单次查询中同时计算出商务舱和经济舱的总余票。

**2. 为什么这么实现？**

- **数据库算力前置**：很多人会犯一个错误——写一个 `SELECT *` 把十万条数据全部拉到 Java 内存里，然后用一个 `while` 循环或者 Java 8 Stream 去累加。这极其消耗网络 I/O 和内存。你在这里直接利用了数据库底层 C/C++ 写好的聚合引擎，让数据库算好结果只返回一行数据，这是**性能最高**的做法。

**3. 替代方案：**

- 如果查询极其频繁且历史航班的余票不会再变，可以在数据库里建立一个名为 `flight_daily_summary` 的**物化视图 (Materialized View)**，每天定时刷新，查询时直接读取视图，达到毫秒级响应。

------

### 模块二：`searchFlights` (多表联合检索)

**1. 逻辑实现：**

执行了一个跨越 5 张表的 `JOIN`，并在 `WHERE` 语句中使用了 `(src.city ILIKE ? OR src.iata_code ILIKE ?)`。

**2. 为什么这么实现？**

- **高度规范化的代价**：因为数据库设计遵循了第三范式（3NF），消除了冗余，所以航班、航线、机场信息散落在不同的表里，必须通过外键重新拼装。
- **用户体验 (UX)**：使用 `ILIKE`（PostgreSQL 特有的不区分大小写匹配）和 `OR` 逻辑，允许用户输入“Beijing”、“beijing”或“PEK”，极大提升了系统的容错率。

**3. 替代方案：**

- **全文搜索引擎**：如果数据量达到千万级，5 表 JOIN 的性能会断崖式下降。真实业务中（如携程、飞猪），复杂的航班搜索通常会通过同步工具（如 Canal）把组装好的数据异构到 **Elasticsearch (ES)** 中。用户搜索时直接打在 ES 索引上，而不是硬查关系型数据库。

### 模块三：`manageOrders` (增删查改与事务核心)

这是整套代码的“灵魂”所在，体现了对 **ACID（原子性、一致性、隔离性、持久性）**的深刻理解。

#### 3.1 `bookTicket` (买票 - 增)

**1. 逻辑实现：**

关闭了自动提交（`setAutoCommit(false)`），第一步通过 `UPDATE ... WHERE remain_count > 0` 扣减库存，第二步 `INSERT` 订单，最后 `commit()`。

**2. 为什么这么实现？**

- **防超卖（乐观悲观锁结合）**：利用 `WHERE remain_count > 0` 作为更新条件，这在底层触发了行级排他锁（Row-level Exclusive Lock）。当 10 个人同时抢最后 1 张票时，数据库会自动将他们排队。第一个人扣减成功，库存变 0；后面 9 个人执行时因为条件不满足，`executeUpdate()` 返回 0，程序立刻 `rollback()`，完美防止了库存变成负数。

**3. 替代方案：**

- 在“双十一”或“春运”这种高并发场景下，直接打库依然会把数据库连接池撑爆。工业界通常会把库存预热到 **Redis** 中，利用 Redis 的 Lua 脚本进行原子扣减，扣减成功后再丢到 **RabbitMQ/Kafka** 消息队列里，让数据库慢慢“消化”生成订单。

#### 3.2 `cancelOrder` (退票 - 改)

**1. 逻辑实现：**

同样使用事务。亮点在于使用了 `UPDATE ... RETURNING flight_id, cabin_class`。

**2. 为什么这么实现？**

- **减少网络往返 (Network Round-trips)**：常规做法是先 `SELECT` 查出订单属于哪个航班什么舱位，然后再 `UPDATE` 状态。使用 PostgreSQL 的 `RETURNING` 魔法语法，可以在更新状态的瞬间把你需要的值“顺手”带回来，省去了一次数据库查询，性能加倍。

#### 3.3 `deleteOrder` (物理删除 - 删)

**1. 逻辑实现：**

执行 `DELETE FROM ticket_order WHERE order_id = ?`。

**2. 为什么这么实现？**

- 最直接的物理抹除，释放磁盘空间。

**3. 替代方案（非常重要）：**

- **逻辑删除 (Soft Delete)**：在真正的商业系统中，**我们几乎永远不会执行 `DELETE` 语句**（尤其是订单和流水等核心数据）。通常的做法是在表里加一个布尔字段 `is_deleted DEFAULT false`。所谓的删除，只是执行 `UPDATE ticket_order SET is_deleted = true`。这样不仅查询时可以过滤掉它，还能保留完整的数据用于后续的财务对账、大数据分析或客诉追溯。
