# Linux实现

## 1. 实验环境配置 (Environment Setup)

Windows 11 (Client) 与 WSL2 Ubuntu 24.04 (Server) 跨系统协作。

### 服务端环境 (Server - Linux/WSL2)

- **操作系统：** Ubuntu 24.04 LTS
- **数据库：** PostgreSQL 16.13
- **监听端口：** **5433** (为避开 Windows 宿主机默认 5432 端口冲突而自定义)

------

## 2. Linux 数据库部署与表结构设计

### 2.1 数据库初始化

在 Linux 终端执行了以下核心操作以打通连接通道：

1. **服务管理：** 使用 `sudo service postgresql start` 启动引擎。
2. **安全授权：** 通过 `psql` 强制修改 `postgres` 超级用户密码为 `123456`。
3. **网络配置：** 在 `postgresql.conf` 中将端口改为 `5433` 并确保其监听本地流量。

### 2.2 Schema 优化与脏数据处理

针对民航原始 CSV 数据的特点，对 DDL 进行了两项关键调整：

- **自增主键适配：** 弃用 MySQL 的 `AUTO_INCREMENT`，采用 PostgreSQL 原生的 `BIGSERIAL` 类型，确保在高并发插入下的 ID 生成效率。
- **容错设计：** 针对 `clean_airport.csv` 中部分支线机场（如荔波机场 LLB）缺失海拔数据的问题，动态移除 `altitude` 字段的 `NOT NULL` 约束，避免事务级联回滚。

------

## 3. 高性能导入策略 (Performance Optimization)

针对初始导入耗时较长（约 69.3 秒）的问题，在 Java 应用层实施了“三位一体”优化。

### 3.1 协议层重写 (Protocol Rewrite)

在 JDBC URL 中注入 `reWriteBatchedInserts=true`。

- **原理：** 驱动程序将成千上万条独立的 `INSERT` 自动拼接为一条多值插入语句，极大降低了 Windows 与 WSL 之间的网络往返次数（RTT）。

### 3.2 事务手动控制 (Transaction Control)

显式调用 `con.setAutoCommit(false)`，并在单表或全流程结束后统一 `con.commit()`。

- **原理：** 绕过数据库默认的“逐行刷盘”机制，消除磁盘 `fsync` 的高频 I/O 阻塞。

### 3.3 批量发车频率 (Batch Sizing)

设定每 10,000 条记录执行一次 `executeBatch()`。

- **原理：** 在吞吐量最大化与 JVM 堆内存占用之间取得最佳平衡。

------

## 4. 性能基准测试结果 (Benchmarks)

经过优化后，导入近 **35 万条** 数据的性能提升对比显著：

| **数据表 (Entity)** | **数据量 (Records)** | **初始耗时 (无优化)** | **优化后耗时 (本次)** | **性能提升倍数** |
| ------------------- | -------------------- | --------------------- | --------------------- | ---------------- |
| `route`             | 21,764               | 4,875 ms              | **443 ms**            | ~ 11 倍          |
| `flight`            | 108,820              | 21,102 ms             | **1,046 ms**          | ~ 20 倍          |
| `flight_cabin`      | 217,640              | 43,309 ms             | **2,336 ms**          | ~ 18 倍          |
| **总计吞吐量**      | **~ 349,555 条**     | **~ 69.3 秒**         | **~ 3.98 秒**         | **~ 17.4 倍**    |

**结论：** 优化后的数据管线实现了约 **87,700 TPS** 的吞吐率，成功解决了跨操作系统 I/O 瓶颈，系统表现稳健。
